Stage 4：Reranker

Stage 4 是最后一道筛选层。Stage 3 可以多拿候选，但真正进入系统 prompt 的记忆必须少、准、可解释。Reranker 会融合规则信号、source prior 和 Stage 3 evidence，选出最终记忆。

从候选到上下文

它回答的问题是：哪些记忆最终进入 prompt。

心智模型

把 Reranker 想成“带业务规则的最后一道闸”。
Filter

先排除明显不该进来的

如果候选和当前用户请求冲突，或者没有 domain、keyword、destination、applicability 任何相关性，就先过滤。

Score

再把规则分和证据分合起来

规则分来自 bucket、domain、keyword、destination、recency、applicability、conflict；证据分来自 Stage 3 lanes。

Budget

最后按 source 预算挑选

profile 和 episode slice 分开归一化，再按 retrieval source 决定选几个，防止某一来源天然压倒另一来源。

Stage 4 不调用 LLM。它是确定性 reranker，核心价值是可解释、可测试、可通过 telemetry 复盘。

伪代码

对应 choose_reranker_path() 的主路径。
def choose_reranker_path(candidates, user_message, plan, retrieval_plan, evidence_by_id, config):
    if not candidates:
        return empty_rerank_result()

    intent_profile = resolve_intent_profile(user_message, retrieval_plan)
    intent_label = resolve_intent_label(user_message, retrieval_plan)

    normalize Stage3 fused / lexical / semantic scores across candidates

    profile_scored = []
    slice_scored = []
    for candidate in candidates:
        detail = compute_rule_signals(candidate, user_message, plan, retrieval_plan)
        detail.rule_score = weighted_sum(
            bucket, domain, keyword, destination, recency, applicability, -conflict
        )
        detail.evidence_score = weighted_sum(
            symbolic_hit, lexical_hit, semantic_hit,
            lane_fused_score, lexical_score, semantic_score, destination_match_type
        )

        if conflict_score >= 0.95:
            mark dropped as conflict
            continue
        if has no domain/keyword/destination/applicability signal:
            mark dropped as weak_relevance
            continue

        split into profile_scored or slice_scored

    dedupe profile and slice candidates separately

    if remaining_total <= small_candidate_set_threshold:
        selected = select_by_source_budget_without_weighted_sort()
        return fallback_used=\"skipped_small_candidate_set\"

    normalize profile scores and slice scores separately
    selected = select_by_source_budget(profile, slices, retrieval_plan.source)
    return selected candidates with per-item reasons and structured scores

评分组成

不是单纯向量分数，而是 source-aware weighted rerank。
信号
包含什么
为什么需要
rule_score
bucket prior、domain/keyword overlap、destination、recency、applicability、conflict。
保留旅行业务规则：约束和拒绝项通常比弱偏好更重要。
evidence_score
lane_fused_score、semantic_score、lexical_score 等 Stage 3 证据。
让 symbolic、semantic、lexical 的命中强度影响排序。
source_normalized_score
profile / slice 分源 min-max 归一化。
避免某一类来源天然分数尺度压倒另一类。
final_score
归一化后加 source prior。
用于最终 source budget selection。

规则信号

业务相关性先由确定性规则表达出来。
信号
计算方式
设计理由
bucket_score
profile bucket prior：constraints 最高，其次 rejections、stable_preferences、preference_hypotheses。
硬约束和明确拒绝通常比弱偏好更重要。
domain_exact_score
retrieval plan domains 与 candidate domains 的 Jaccard。
保证记忆领域和当前请求对齐。
keyword_exact_score
plan keywords 与 candidate terms 的 Jaccard。
防止只靠语义相似引入泛化噪声。
destination_score
retrieval destination 或当前 plan destination 是否出现在候选摘要 / applicability。
旅行记忆强依赖目的地，不同城市经验不一定可迁移。
recency_score
按创建时间做半衰期衰减，默认 180 天。
近期偏好和踩坑通常更可信。
applicability_score
匹配 destination、家庭/儿童、父母/老人、少走路等适用范围。
旅行约束经常和同行人、体力、目的地条件绑定。
conflict_score
当前用户消息和候选 polarity / bucket 冲突时置高。
避免把用户这轮明确否定的旧偏好注入 prompt。

Intent Weight Profile

不同意图下 profile 和 slice 的先验不同。
profile

画像优先

当 retrieval source 是 profile 或 reason 含 profile 信号时，profile_source_prior 更高，用于偏好、约束、拒绝项。

episode_slice

历史切片优先

当 source 是 episode_slice 或 reason 指向 past trip 时，slice_source_prior 更高，用于过去行程体验、踩坑、已验证安排。

recommend

推荐场景平衡

用户说“推荐”“比较好”“适合我”“怎么安排”时，profile 和 slice 都可能有价值，权重更均衡。

硬过滤和去重

不是所有候选都应该参与最终排序。
HARD FILTER

冲突过滤

当 conflict_score >= 0.95 时直接丢弃。例如候选是“喜欢青旅”，但用户当前说“不住青旅”。

弱相关过滤

如果 domain、keyword、destination 都没有命中，且 applicability 也很弱，则标记为 weak_relevance 并丢弃。

DEDUPE

Profile 去重

按 source、primary domain、key、polarity 组成 duplicate group，避免同一偏好重复注入。

Slice 去重

按 source、bucket、primary domain、summary tokens 组成 duplicate group，避免相似历史切片重复占用上下文。

选择策略

最后一步不是单纯全局 top N。
retrieval source
选择方式
为什么这样做
profile
优先选 profile 前 profile_top_n=4；如果没有，再用 slice 兜底。
偏好/约束问题应优先注入长期画像。
episode_slice
优先选 slice 前 slice_top_n=3；如果没有，再用 profile 兜底。
历史经验问题应优先注入过去行程切片。
hybrid_history
先取 profile 前 2 和 slice 前 2，再跨 source 排序，最多 hybrid_top_n=4。
混合场景下避免某一类来源把另一类完全挤掉。
小候选集
剩余候选数 <= 3 时跳过 weighted rerank。
候选很少时排序收益有限，直接保留更简单，也保留 trace 标记。

为什么旅行规划需要 Reranker

召回太多会污染规划，召回太少会失去个性化。
Constraints

硬约束优先

“不吃辣”“不坐红眼”“老人少走路”这类约束必须比普通喜好更靠前。

Conflict

避免冲突记忆

用户当前说“这次别按以前习惯”，旧偏好不能无脑注入，需要 conflict penalty。

Budget

控制 prompt 预算

旅行状态本身很长，记忆只能精选。Reranker 把候选压缩成少量高价值上下文。

Stage 4 有效的关键是“先允许 Stage 3 召回得稍宽，再用可解释的业务排序收窄”。这比一开始就只做窄检索更能兼顾召回率和精准度。
