问题阐述：
在使用本项目时，我发现了如下的一些问题。我的表达可能有时候觉得混乱，你应该从我的表达中总结出其核心观点。

> **状态说明**：以下问题已在 `feature/prompt-architecture-upgrade` 分支中通过 skill-card 提示词架构升级全部解决。

1. ✅ **[已解决]** 第一阶段时，主要目标是确定目的地。但在实际和旅行 Agent 进行交互的过程中，我发现当用户提出一些问题时，Agent 的回答内容常包含大量的无关内容，可能淹没用户的焦点。同时，Agent 给出意见之前并不能保证都有足够的外部信息的支撑。
   - **方案**：Phase 1 重写为 skill-card，增加"先查后说"硬规则（每条建议须有工具查询结果支撑）和回复简洁度约束（≤150字）。

2. ✅ **[已解决]** 第三阶段时，存在两个问题。当前的流程把大交通和住宿放在最后确认。以我个人旅行的体验是，一般会确定旅行的时间，在旅行确定时间之后会确定大交通。当确定完大交通之后，就可以把注意力完全放在旅行内容的规划上。
   - **方案**：Phase 3 拆分为 4 个独立子阶段 prompt（brief/candidate/skeleton/lock），lock 子阶段独立处理大交通时序——确认时间后即锁定大交通。

3. ✅ **[已解决]** 在写入旅行计划的框架时，当前的过程会试图一次性更新所有内容，可能会导致内容过长，应该采用按天多次更新的方式。
   - **方案**：Phase 5 从"优先一次性用 list[dict] 提交全部天数"改为"按1-2天增量调用 assemble_day_plan"。

4. ✅ **[已解决]** 旅行框架的搭建，需要有专门的提示词，因为每一天如何安排，是需要进行反复思考权衡的。
   - **方案**：skeleton 子阶段增加"锚定→约束→分组→取舍"结构化思考框架。

5. ✅ **[已解决]** 在第五个阶段对每一天的旅行内容进行规划，其实更多的是一个路径规划问题，现在实际执行的提示词，我不知道有没有完全按照这个核心的目标进行设计。
   - **方案**：Phase 5 重新定位为路径规划优化问题（最小化无效移动，最大化体验密度），prompt 围绕路线优化核心设计。

在第三步旅行骨架搭建的过程当中，这一步在人类规划旅行的时候，是一个非常好非常耗时的过程。

它涉及到目的地的一些筛选。这个过程之前会锚定一些不变量，根据这些不变量，然后再综合各种用户分享，才能够给出最后的决断。

6. ✅ **[已解决]** 但是目前在进行规划的时候，我发现 agent 并没有实现一个非常科学的框架搭建的前置的信息收集安排的过程。
   - **方案**：Phase 3 base prompt 增加结构化信息收集流程（先锚定不变量，再综合搜索信息，最后决策）。

7. ✅ **[已解决]** 当前第一阶段存在越界行为，可能会因为 Agent 的引导导致用户输出所有信息。
   - **方案**：Phase 1 增加 Red Flags 强化阶段边界（"不提前进入行程规划"、"不主动引导用户输出所有信息"），GLOBAL_RED_FLAGS 跨阶段注入。

8. ✅ **[已解决]** 现在的第三阶段又存在不收敛的问题。
   - **方案**：Phase 3 从全量加载改为按子阶段动态拼装 prompt（`build_phase3_prompt(step)`），每个子阶段只看到自己的规则和完成 Gate；brief 子阶段增加收敛压力（≤2轮完成）。
