# 记忆系统评审：旅行 Agent 的记忆到底该怎么搞？

> 本文档最初写于 2026-04-16，后经多次讨论修订。
>
> 修订重点：把关注点从"召回端怎么检索更准"转向"写入端怎么记才对"，并补充了若干原文遗漏但旅行场景下很关键的问题。
>
> 写给大一同学：本文会从"代码现在怎么做的"开始讲，再用旅行场景的例子解释问题在哪，最后给出修法。遇到术语我会尽量解释。

## 先说结论

这套记忆系统的方向是对的——它已经有了结构化记忆、候选提取、风险分类、trip/global 作用域、episode 归档这些关键零件。但核心短板不在"怎么检索更准"，而在"怎么写入才对"。

具体来说，三个最根本的问题：

1. **写入时丢了语境**——把"带老人去黄山那次想要慢节奏"记成"喜欢慢节奏"，下次一个人去上海也给你安排慢节奏
2. **一次就当永久**——用户只在一次旅行里表现出来的习惯，直接写进了永久记忆，实际上可能是那次的偶然
3. **否决和偏好没区分**——"我绝不住青旅"和"我比较喜欢民宿"是两种东西，前者跨所有场景都成立，后者换个情境可能就变了

如果先把这三件事理对，哪怕召回端只做最简单的规则注入，效果也会有质的提升。反过来，写入不对，召回端做再花哨也救不回来。

## 基础知识：这套记忆系统是干什么的？

想象你是一个旅行顾问。你有一个笔记本，上面记着客户的各种信息：

- "这个客户一般预算 1 万左右"（全局偏好）
- "这次带老人，节奏要慢"（本次旅行约束）
- "上次去云南选了自由行，整体满意"（历史经验）

这套系统做的就是：**帮你自动从对话里提取这些信息，存起来，下次聊天时又自动找出来，喂给 AI，让它记住这个人**。

存储格式叫 `MemoryItem`，大概长这样：

```
id:           mem_abc123
type:         preference        ← 偏好类型（偏好/否决/约束）
domain:       pace              ← 属于哪个领域（节奏/餐饮/酒店...）
key:          pace_preference    ← 偏好键名
value:        "慢节奏"           ← 偏好值
scope:        trip              ← 作用域（global=跨旅行 / trip=仅本次）
confidence:   0.8               ← 置信度（0~1）
status:       active            ← 当前状态
```

另外还有一种叫 `TripEpisode`，是一次完整旅行结束后的归档摘要：

```
destination:      "黄山"
dates:            "2026-05-01 - 2026-05-03"
travelers:        {adults: 2, seniors: 1}
selected_skeleton:{id: "sk_001", ...}    ← 最终选了哪个方案
accepted_items:   [...]                  ← 这次接受了哪些偏好
rejected_items:   [...]                  ← 这次否决了哪些偏好
lessons:          ["雨天备选室内景点"]      ← 经验教训
satisfaction:     null                   ← 满意度（还没填）
```

下面的每个问题，都会讲：代码怎么做的 → 旅行场景里哪里不对 → 怎么改。

## 问题 1：TripEpisode 写了但没用上

### 代码怎么做的

系统在一次旅行结束时，会把这次旅行的完整信息打包成一个 `TripEpisode`，存进用户的 `trip_episodes.jsonl` 文件。

但在给 AI 构造记忆上下文时（`MemoryManager.generate_context()`），系统只从 `MemoryItem` 里找记忆，包括三路：

- `retrieve_core_profile`：找全局（`scope == global`）的活跃记忆
- `retrieve_trip_memory`：找当前旅行（`scope == trip`）的记忆
- `retrieve_phase_relevant`：找跟当前阶段（Phase 1/3/5/7）相关的记忆

这三路的输入全是 `MemoryItem`，**TripEpisode 完全没有参与召回**。

### 问题在哪

打个比方：你每次帮客户做完一次旅行，都认真写了总结报告，但下次这个客户再来找你规划新旅行时，你从来不翻之前的总结报告。

对一个旅行 Agent 来说，episode 不是"锦上添花"，而是最接近真实旅行经验的资产。用户第二次来规划时，真正有价值的往往不是某个离散的偏好键值对，而是：

- 上次去了哪里，满意还是不满意
- 上次最后选了怎样的旅行骨架方案
- 上次踩了哪些坑，这次应不应该主动避开

现在的问题就是：系统花了精力存了这些经验，但 AI 看不到它们。

### 怎么改

**最小改动方案**（推荐先做这个）：

在 `MemoryManager.generate_context()` 里加一步：调用 `list_episodes(user_id)`，取最近 3 条、目的地相同或最近完成的 episode，格式化成"历史旅行经验"区块，注入 system prompt。

这改动量很小，但能让 AI 突然"记得上次旅行发生了什么"。

**后续优化**：

- 让 episode 进入主召回闭环，按目的地、旅行天数、同行人类型、预算段等维度做检索
- 引入 episode 摘要压缩，只保留对当前阶段有帮助的经验

## 问题 2：记忆写入时丢了语境，这是最根本的问题

### 代码怎么做的

当用户发送消息后，系统会从对话里提取记忆。提取过程在 `backend/main.py` 的 `_extract_memory_candidates()` 里：

1. 从最近 8 条用户消息里筛选出内容
2. 把这些消息、当前行程事实、已有记忆条目拼成一个 prompt，交给 LLM
3. LLM 返回结构化的候选记忆（MemoryCandidate）
4. 经过策略审核（MemoryPolicy）后写入

提取结果就是一条条 `MemoryItem`，比如：

```
key: pace_preference, value: "慢节奏", domain: pace, scope: trip
```

注意看：提取的时候，原始对话的语境全丢了。用户是在"带老人、黄山、3 天、雨天"的背景下说"节奏太快了"，但提取出来就只剩一个"慢节奏"。

### 问题在哪

这是整个记忆系统**最根本的设计缺陷**，但原文档没把它当成最核心的问题。

旅行规划是一个高度依赖情境的领域。同一句话，在不同的旅行背景下意思完全不同：

| 原始语境 | 用户说的话 | 提取成无语境记忆后 |
|----------|-----------|-----------------|
| 带老人去黄山3天 | "节奏太赶了" | pace: 慢节奏 → 下次一个人去上海也慢？ |
| 雨天备选室内景点 | "不想纯逛景点" | dislike: 景点 → 晴天城市也避开景点？ |
| 预算5000去昆明 | "住便宜点" | hotel: 经济型 → 商务出差也安排经济型？ |
| 情侣度假 | "想要浪漫一点" | style: 浪漫 → 带父母出游也安排浪漫？ |

脱离了语境的记忆不是"不够精准"，而是**会产生主动误导**。一条在黄山带老人背景下有效的记忆，原封不动搬到上海出差的场景里，效果比没有这条记忆更差。

原文档第 3 点的修复方向是"把 assistant 的提议和用户的行为也喂给提取器"，但这只是扩大了输入范围，**没有改变输出依然是去语境化 key-value 对**的问题。喂更多上下文进去，提取出来的还是一条缺乏情境锚定的记录。

### 怎么改

**核心思路：记忆写入时必须带上原始语境**

```python
# 现在：只有 key-value
MemoryItem(key="pace_preference", value="慢节奏", domain="pace")

# 应该：带语境标签
MemoryItem(
    key="pace_preference",
    value="慢节奏",
    domain="pace",
    context="带老人/黄山/3天/雨天",       # ← 原始语境
    applicability="narrow",               # ← 适用范围宽窄
    travel_type=["family", "leisure"],     # ← 旅行类型标签
)
```

在提取 prompt 里就要要求 LLM：
1. 判断这条偏好是**稳定偏好**（如"不吃辣"）还是**情境性反应**（如"这次节奏太赶"）
2. 如果是情境性的，必须保留原始上下文
3. 召回时，先匹配语境相似度，而不是只匹配 domain

## 问题 3：一次观察就写入永久记忆，缺乏稳定性验证

### 代码怎么做的

当前的记忆提取和写入链路是这样的：

```
用户说了什么 → LLM提取 → MemoryPolicy分类 → auto_save → 永久写入
```

`MemoryPolicy` 的分类逻辑（`backend/memory/policy.py`）：

- 如果 domain 是 `payment`、`membership` → 直接丢弃
- 如果包含身份证号、护照号等 PII → 直接丢弃
- 如果风险是 `high` → 需要人工审核（pending）
- 如果风险是 `low` 且置信度 >= 0.7 → 自动保存（auto_save）
- 如果风险是 `medium` 且置信度 >= 0.8 → 自动保存

也就是说，**只要 LLM 提取时给了足够的置信度，一次观察就能直接写入永久记忆**。

### 问题在哪

用户的行为在单次旅行里是被各种偶然因素严重污染的：

- 第一次选了经济型酒店 → 写入 `hotel_preference: 经济型`
- 带老人那次选了慢节奏 → 写入 `pace: 慢`
- 某次雨天抱怨景点无聊 → 写入 `dislike: 纯景点`

这些都是**情境性反应**，不是**稳定偏好**。但当前系统把它们同等对待，直接写入全局记忆。

这和问题 2 叠加后伤害最大：**写入时丢了语境，又把情境性反应当作稳定偏好永久保存**。

合理的模型应该是"假设-确认模型"：

| 层级 | 写入条件 | 生命周期 | 例子 |
|------|---------|---------|------|
| trip 事实 | 单次行程中提取 | 随 trip 过期 | "这次带老人，节奏要慢" |
| 待确认偏好 | 出现 1 次，暂不写入全局 | pending，不参与主召回 | "似乎偏好慢节奏（仅观测 1 次，待确认）" |
| 稳定偏好 | 跨 2-3 次旅行一致出现 | 永久，参与主召回 | "确实偏好慢节奏（已确认 3 次）" |

当前系统有 `pending` 状态，但它的含义是"等用户确认该不该保存"，不是"等多次观察确认这是不是稳定偏好"。`confidence` 字段是"提取置信度"（LLLM 觉得自己提取得对不对），不是"稳定性置信度"（这条偏好是不是跨场景成立的）。这两个概念被混在一起了。

### 怎么改

**最小改动方案**：

1. 新增一个字段 `stability`，表示这条记忆被不同旅行确认了几次
2. `auto_save` 低风险记忆时，如果是 `scope == global`，初始 `stability == 1`，status 标记为 `pending`
3. 只有当同一条偏好在第 2 次及以上旅行中出现时，`stability += 1`，status 才升为 `active`
4. 追加一条规则：`global + stability == 1` 的记忆只以低权重参与召回，且在 prompt 里注明"仅观测 1 次，待确认"

**这是 TripEpisode 的另一个关键价值**：它不只是归档，而是偏好的验证场。每次新行程结束后的 TripEpisode，都应该拿去跟 pending 的全局偏好做比对，确认或推翻。

## 问题 4：否决信号比偏好信号更可信，但系统同等对待

### 代码怎么做的

当前 `MemoryItem` 有一个 `type` 字段，可以取值 `preference`（偏好）、`constraint`（约束）、`rejection`（否决）。在召回排序时：

```python
# backend/memory/retriever.py 中的排序逻辑
def _sort_key(item):
    type_order = {"constraint": 0, "rejection": 0, "preference": 1}
    return (type_order.get(item.type, 1), -item.confidence, -item.updated_at, item.id)
```

否决和约束排在偏好前面，看起来有区分。但它们的**生命周期、写入门槛、召回权重、过期策略**完全一样。唯一的区别就是排序时否决排前面。

### 问题在哪

对旅行 Agent 来说，否决和偏好的可靠性差距非常大：

- "我绝不住青旅" → 跨几乎所有场景都成立，几乎不需要语境
- "我比较喜欢民宿" → 换个情境可能就变了，预算高的时候可能想住五星

用日常类比：

- 否决 = 食物过敏（"我对花生过敏"），你需要100%记住，永远不过期
- 偏好 = 口味偏好（"我喜欢川菜"），大多数时候成立，但偶尔也会想吃清淡的

如果把这两者放在同一条管线上同等对待，就会出现两种问题：

1. 否决没有得到应有的重视——"绝不住青旅"和"比较喜欢民宿"在系统里本质上是同一种东西，记忆校验、过期策略完全一样
2. 偏好又得到了不应有的信任——一次说"喜欢慢节奏"就跟"绝不住青旅"一样永不过期

### 怎么改

**核心思路：否决和偏好走不同的管线**

| 维度 | 否决/约束 | 偏好 |
|------|----------|------|
| 写入门槛 | 低，一次就写入 | 高，需多次确认 |
| 生命周期 | 长期有效 | 随情境衰减 |
| 召回权重 | 高，始终注入 | 低，仅参考 |
| 过期策略 | 只有人工删除 | 长期未确认则降级 |

最小改动：先把 `constraint` 和 `rejection` 类型从稳定性验证流程里豁免——它们不需要等待多次确认，应该直接写入并高权重召回。同时把 `preference` 类型纳入问题 3 中的稳定性验证流程。

## 问题 5：旅行类型是记忆的第一正交维度，但系统缺失

### 代码怎么做的

当前 `MemoryItem` 的 `scope` 字段只有两个值：

- `global`：跨所有旅行通用
- `trip`：仅当前旅行有效

### 问题在哪

同一个人在不同类型的旅行里，行为模式几乎不相关：

- 出差 2 天 vs 全家国庆 7 天游 → 酒店偏好、节奏、预算完全不同
- 一个人短途 vs 情侣度假 → 住宿类型、活动偏好完全不同
- 国内周末 vs 国际长线 → 风格、预算、期望完全不同

"带老人节奏要慢"这条记忆，只有在"亲子/休闲"类型的旅行里有参考价值。如果不加旅行类型标签，召回时要么全注入（污染上下文），要么按关键词碰运气（不可靠）。

更根本的是，**`global` 这个作用域太粗暴了**。很少有偏好是真正"全局"的——"不吃辣"可能算一个，"喜欢慢节奏"绝对不算。而 `trip` 又太窄了，只限当前旅行。中间缺少一个维度：**同类旅行类型**。

### 怎么改

**最小改动方案**：

在 `MemoryItem` 里增加 `travel_type` 标签（出差/休闲亲子/情侣/独行/...），提取时就标注。召回时先过滤旅行类型匹配的记忆，再按 domain 和置信度排序。

`global` 只用于真正的跨类型稳定偏好（如饮食过敏、绝对否决项），其他偏好降级为"同类类型内有效"。

## 问题 6：记忆没有闭环校验

### 代码怎么做的

当前记忆是单向流：

```
提取 → 写入 → 召回
```

写入之后就再也没机制验证这条记忆有没有帮到用户。用户上次说"喜欢自然风光"，这次推荐了自然风光的路线，用户拒绝了——这个拒绝信号不会回过头去修正或降级那条偏好记忆。

### 问题在哪

用科学研究打个比方：你提出了一个假设（"用户喜欢自然风光"），做了实验（推荐自然风光路线），观察到了结果（用户拒绝了），但你从来不根据实验结果修正假设。

对旅行 Agent 来说，**每次新行程的决策结果都是对旧记忆的校验机会**：

- 用户最终选了什么 → 确认或修正已有偏好
- 用户否了什么 → 否决信号或推翻旧假设
- 用户跳过了系统推荐的选项 → 这条记忆可能不够可靠

如果没有这个闭环，错误记忆会永远留在系统里，越来越多，越来越脏。

### 怎么改

**核心思路：每次行程结束后，用实际决策结果校验已有记忆**

这个过程在 `_build_trip_episode()` 里已经搭好了框架——它已经在收集 `accepted_items` 和 `rejected_items`。只需要增加一步：

1. 对比本次行程的实际决策和已有全局偏好的方向是否一致
2. 一致的 → `stability` +1（增强置信度）
3. 不一致的 → 降级为 `pending` 或标注冲突
4. 被明确否决的偏好 → 直接标为 `obsolete`

这样可以形成闭环：**提取 → 写入 → 召回 → 实际决策 → 校验 → 修正**，而不是现在的单行道。

## 问题 7：记忆提取输入只看用户消息，太窄

### 代码怎么做的

提取链路在 `_extract_memory_candidates()` 里：

1. 从 `messages_snapshot` 筛出最近几条 `role == USER` 的消息
2. 默认只取最近 8 条（配置在 `backend/config.py`）
3. 把用户消息、行程事实、已有记忆拼成 prompt
4. 交给 LLM 提取候选记忆

这意味着：assistant 的回复、工具调用结果、用户对候选方案的接受/否定方式，都不会直接进入提取 prompt。

### 问题在哪

用户的稳定偏好经常不是一句话说出来的，而是通过行为表现出来的：

- 连续拒绝了所有红眼航班选项 → 偏好 "不坐红眼"
- 总是选择带早餐的酒店 → 偏好 "含早"
- 对某类方案反复点否定 → 否决信号

这些信号存在于 assistant 提议 + 用户反应的交互中，不是用户原话。只看用户原话，会错过大量"偏好是通过选择行为表达的"信息。

但需要注意：**这是第二优先级的问题**。在问题 2（写入时保留语境）和问题 3（稳定性验证）解决之前，扩大提取输入范围只会引入更多噪声。应该先把提取质量做对，再扩大输入。

### 怎么改

**推荐顺序**：

1. 先解决问题 2（语境保留）和问题 3（稳定性验证）
2. 再把最近几轮中与用户决策直接相关的 assistant 内容加入提取 prompt
3. 当用户出现明显否定或确认时，把那一轮前后的 assistant 提议一起交给提取模型

**暂不建议**把所有消息都塞进提取 prompt——旅行场景对话很长，全塞进去成本高、噪声大。只塞与决策相关的部分即可。

## 问题 8：记忆召回以规则过滤为主，缺少语义能力

### 代码怎么做的

记忆召回由 `backend/memory/retriever.py` 负责，三路检索：

1. `retrieve_core_profile`：找 `scope == global` 且 `status == active` 的核心画像，限 10 条
2. `retrieve_trip_memory`：找 `scope == trip` 且 trip_id 匹配的记忆
3. `retrieve_phase_relevant`：根据当前阶段白名单筛选 domain，限 8 条

阶段白名单（`_PHASE_DOMAINS`）：
- Phase 1（需求收集）：destination, pace, budget, family, planning_style
- Phase 3（方案选择）：+ hotel, flight, train, accessibility
- Phase 5（细排）：pace, food, accessibility, family, budget
- Phase 7（最终确认）：documents, flight, train, food, accessibility

排序规则：constraint/rejection 优先 → confidence 降序 → updated_at 降序。

也就是说，**当前召回不是"语义上最相关的记忆"，而是"规则过滤后最像应该出现的记忆"**。

### 问题在哪

规则检索有两个天然问题：

1. **依赖 domain 分类准确**：如果提取时把一个关键偏好放进了 `general`，后面各阶段可能根本召回不到
2. **不会理解语义相似性**：用户说"不喜欢人太多的地方"，提取成 `general/crowd_preference`，在做节奏规划、区域筛选、景点推荐时，这条记忆其实很重要，但规则召回不一定命中

但要注意：**在当前项目阶段，语义召回是过度设计**。一个用户通常只有 20-50 条记忆，对 50 条记忆做 embedding 检索和混合排序，边际收益很低。更实际的做法是：

### 怎么改

**最小改动方案**（推荐先做）：

1. 扩充 `_PHASE_DOMAINS`，让 Phase 5 也包含 `destination`（有时在细排阶段还需要目的地偏好）
2. 对 `general` 域的高置信度 item 做一次补充扫描，避免重要记忆因分类保守而永久沉底
3. 增加 keyword 软匹配：除了 domain 精确命中，还允许基于 key/value 做关键词匹配

**后续优化**（当记忆量增长到数百条时）：

- 引入向量化语义检索，建立 embedding 索引
- 做混合检索：规则过滤做安全边界，embedding 相似度做相关性排序

## 问题 9：trip 作用域的记忆没有过期机制

### 代码怎么做的

`MemoryItem` 有一个 `expires_at` 字段：

```python
expires_at: str | None = None
```

但当前代码中：
- `MemoryPolicy.to_item()` 没有给它赋值
- `FileMemoryStore.list_items()` 没有过滤过期项
- `MemoryRetriever` 的三类检索也没有检查过期
- 没有后台清理或懒清理逻辑

这个字段目前只是 schema 预留位，没有实际生效。

### 问题在哪

和原文档的观点不同，我认为这不是"生命周期治理未完善"的小问题，而是一个**正确性bug**。

对旅行 Agent 来说，很多记忆天然有时效性：

- "这次带老人" → 不等于以后永远亲子优先
- "这次预算5000" → 不等于以后永远5000
- "这次办签证需要XX材料" → 下次可能政策变了

如果 trip 作用域的记忆在旅行结束后还挂着 `active` 状态，下次规划新旅行时会被召回，造成**跨旅行记忆污染**。这不是锦上添花的优化，是功能正确性问题。

而且它和问题 2 叠加：一条没有语境的 trip 记忆，在新旅行里被错误召回，比没有这条记忆更糟。

### 怎么改

**最小改动方案**：

1. 当旅行结束或新开一个旅行时，把旧 trip_id 下的 `scope == trip` 记忆批量标为 `obsolete`
2. 在召回链路的起点加一步过滤：`status != obsolete`
3. 把 `expires_at` 利用起来，规则可以是：
   - `scope == trip` 的记忆，默认 `expires_at = 旅行结束时间 + 7天`
   - `scope == global` 的偏好，默认不过期，但长期未确认的降级

## 问题 10：新旧模型并存增加维护成本

### 代码怎么做的

项目中同时存在两套记忆数据结构：

旧模型（`UserMemory`, `Rejection`, `TripSummary`）：
```python
class UserMemory:
    explicit_preferences: dict[str, Any]
    implicit_preferences: dict[str, Any]
    trip_history: list[TripSummary]
    rejections: list[Rejection]
```

新模型（`MemoryItem`, `MemoryCandidate`, `TripEpisode`）：
```python
class MemoryItem:
    id: str
    type: str          # preference / constraint / rejection
    domain: str
    key: str
    value: Any
    scope: str         # global / trip
    ...
```

存储层（`backend/memory/store.py`）会兼容读取旧格式，管理层（`backend/memory/manager.py`）也有回退兼容逻辑。另外还有两个同名的 `MemoryMerger` 类（分别在 `extraction.py` 和 `policy.py`），处理的也是两代不同数据结构。

### 问题在哪

这不是立刻会把系统打崩的问题，但会让新同学很难搞清楚哪条才是当前真实链路。做重构时容易误改到兼容层代码。

### 怎么改

**投入最少方案**：

1. 把主链路的 `MemoryMerger` 重命名为 `MemoryItemMerger`
2. 在旧类上加注释：仅供 legacy path 使用
3. 在文档中写清楚当前主路径只依赖 `MemoryItem`

**后续**：

设计一次性迁移脚本，把所有旧格式数据迁移成新格式，然后删除兼容层代码。

## 问题 11：记忆应该围绕决策节点组织，而不是围绕 domain 分类

### 代码怎么做的

当前记忆按 `domain` 分类（budget, hotel, pace, food, ...），召回时也按 domain 白名单过滤。

旅行规划的真实节奏是决策流：

```
选目的地 → 选交通方式 → 选住宿 → 排每天行程 → 确认细节
```

### 问题在哪

用户在选目的地的阶段，需要的不是 `domain == destination` 的所有偏好，而是"上次同类旅行去了哪、满意度如何、有什么教训"。当前按 domain 分类的记忆，和用户的决策流程是脱节的。

这天然指向 TripEpisode："上次同类旅行的整体经验"比"关于目的地的偏好列表"有价值得多。

### 怎么改

**最小改动**：

在问题 1（让 TripEpisode 参与召回）解决后，按决策阶段调整注入策略：

- Phase 1（选目的地）：注入上次同类旅行的目的地 + 满意度
- Phase 3（选方案）：注入上次选了什么类型的骨架 + lessons
- Phase 5（排行程）：注入上次行程节奏/餐饮偏好的 context

长远来看，记忆组织应该从"按 domain 分类"转向"按决策节点 + 旅行类型"交叉组织。

## 其余实况问题

这些问题也存在，但不是当前最优先解决的事。

### PII 防护有基础，但离完善还有距离

当前已有：
- 拒绝 `payment`、`membership` 域
- 正则检测长数字串、邮箱、证件号关键词
- 对候选记忆的 value/attributes/evidence 做脱敏

但对旅行场景的专属敏感信息（航班号、订单号、家庭成员信息、微信号等）覆盖不全，且脱敏只覆盖了候选提取路径，episode lessons 和事件日志等字段还没有脱敏。

### 文件存储扩展性有限，但当前阶段够用

`FileMemoryStore` 全量读写 `memory.json`，不适合高频写入和大用户规模。`asyncio.Lock` 只在单进程内有效。但当前项目阶段用户量小，这些不是立刻要解决的问题。

后续建议先把 `memory.json` 迁移到 SQLite（项目已用了 `aiosqlite`），events 和 episodes 先保持 JSONL。

### ID 截断和事件日志

`MemoryItem` 的 ID 是 SHA1 截断前 16 位，当前规模下冲突概率极低。事件日志只有追加写入，没有读侧分析能力。这些都不是当务之急。

## 优先级总结

按投入产出比排序，最值得先做的：

| 优先级 | 改什么 | 为什么 |
|-------|--------|-------|
| **P0** | trip 记忆过期机制 | 这是正确性 bug，不是优化 |
| **P1** | 写入时保留语境 | 脱离语境的记忆比没有记忆更糟 |
| **P1** | 区分否决和偏好 | 否决一次就写入，偏好需要多次验证 |
| **P1** | 单次观察不能升格为稳定偏好 | 引入 stability 字段和确认流程 |
| **P2** | TripEpisode 参与召回 | 简单注入就能大幅提升"记得上次"的感知 |
| **P2** | 记忆闭环校验 | 每次行程结果反写修正已有记忆 |
| **P3** | 扩大提取输入 | 先把上面的问题解决再扩大输入范围 |
| **P3** | 规则召回补软匹配 | general 域补充扫描 + 关键词匹配 |
| **P4** | 新旧模型清理、PII 补充、存储迁移 | 不急但有技术债 |

**一句话总结**：记忆系统的写入端有三个根本缺陷——去语境化、单次即永久、否决偏好不分。这三件事不改，召回端做再花哨也救不回来。先把写入做对，再做检索做精细。
