# Harness 工程竞争力分析

> **目标**：解释 harness engineering 的业界定义与典型实现，并评估 Travel Agent Pro 当前在 harness 工程上的竞争力、优势和缺口。
>
> **结论先行**：Travel Agent Pro 的竞争力不只来自“能做旅行规划”，而来自一套已经成形的 Agent harness：阶段化运行时、工具边界、质量门控、评估样例、失败分析、可观测性、文档规约和恢复机制。这使项目从普通 LLM Demo 提升为“可被调试、可被评估、可被持续改进”的 Agent 系统。但当前短板也很清楚：eval 运行数据尚未形成持续基线，失败闭环仍偏人工，安全/风险分类还不够系统化，架构约束缺少更多机械化 enforcement。

---

## 1. 业界定义和实现

### 1.1 从传统 test harness 到 Agent harness

在传统软件测试里，ISTQB 对 **test harness** 的定义是：执行测试所需的 stubs 和 drivers 组成的测试环境。这个定义的重点不是测试用例本身，而是“让被测对象能够被驱动、运行、观察和验证的外部装置”。  
来源：[ISTQB Glossary: Test Harness](https://istqb-glossary.page/test-harness/)

Agent 时代的 harness 延续了这个含义，但对象从“函数/模块/系统”变成了“模型 + 工具 + 状态 + 环境”。也就是说，harness 不只是测试框架，而是让模型能够作为 Agent 行动的外部系统。

Anthropic 在 agent evals 文章中给出了两个关键定义：

- **evaluation harness**：端到端运行 eval 的基础设施，负责提供指令和工具、并发运行任务、记录步骤、评分并聚合结果。
- **agent harness / scaffold**：让模型能表现为 Agent 的系统，负责处理输入、编排工具调用、返回结果。评估一个 Agent 时，评估的是模型和 harness 的组合，而不是孤立模型。

来源：[Anthropic: Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)

这一定义对 Travel Agent Pro 很关键：旅行规划能力不是只由 GPT-4o 或 Claude Sonnet 4 决定，而是由“模型 + AgentLoop + 工具系统 + 状态机 + 质量门控 + eval + trace”共同决定。

### 1.2 OpenAI 对 harness engineering 的定义

OpenAI 在 2026 年 2 月的《Harness engineering: leveraging Codex in an agent-first world》中把 harness engineering 明确描述为一种新的工程工作重心：当团队的主要工作不再是手写代码，而是让 Agent 可靠产出时，工程师的任务转向设计环境、表达意图、构建反馈循环。

OpenAI 的实践重点包括：

- repository scaffold、CI、格式化规则、包管理、应用框架；
- `AGENTS.md` 作为目录，而不是巨型说明书；
- structured docs 作为 Agent 可读的 system of record；
- 让 UI、logs、metrics、traces 对 Agent 可见；
- custom linters、structural tests、architecture rules；
- review loop、build failure remediation、eval harness；
- 把人工品味和失败经验沉淀为文档、工具和机械规则。

来源：[OpenAI: Harness engineering](https://openai.com/index/harness-engineering/)

这说明 harness engineering 的核心不是“写更长的 prompt”，而是把环境设计成 Agent 能够理解、验证、修复和持续改进的系统。

### 1.3 Harness 工程的典型能力地图

结合 OpenAI、Anthropic、传统测试和可观测性实践，可以把 Agent harness 拆成 8 类能力：

| 能力 | 目标 | 典型实现 |
|------|------|----------|
| 指令与知识系统 | 让 Agent 知道项目规则和事实来源 | `AGENTS.md`、架构文档、设计文档、索引文档 |
| Runtime scaffold | 让模型能稳定行动 | agent loop、tool calling、状态机、重试、中断、恢复 |
| 工具与环境边界 | 限制 Agent 能做什么、怎么做 | tool schema、权限、沙箱、阶段工具白名单 |
| 质量门控 | 防止错误状态继续扩散 | guardrail、validator、judge、feasibility gate |
| Evaluation harness | 把主观体验变成可运行样例 | golden cases、grader、assertion、pass@k、regression suite |
| Observability | 让运行过程可解释 | trace、logs、metrics、cost、latency、tool timeline |
| 反馈闭环 | 把失败转化为系统改进 | failure taxonomy、postmortem、自动修复、eval 回填 |
| 机械化架构约束 | 防止 Agent 快速产出导致架构漂移 | lint、结构测试、依赖方向检查、schema 校验 |

其中 Anthropic 特别强调：Agent eval 不应只看最终文本，还要看 task、trial、grader、transcript/trace、outcome 和 suite；同时，非确定性 Agent 需要多次 trial，使用 pass@k 或一致性指标理解稳定性。  
来源：[Anthropic: Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)

### 1.4 风险管理和可观测性也是 harness 的组成部分

NIST AI RMF 的目标是帮助组织把可信性考虑纳入 AI 产品、服务和系统的设计、开发、使用和评估。它不是专门讲 Agent harness，但它强调的“设计、开发、使用、评估”全链路风险管理，正好对应 harness engineering 的风险面。  
来源：[NIST AI Risk Management Framework](https://www.nist.gov/itl/ai-risk-management-framework)

OpenTelemetry 对 observability 的定义是：从系统外部理解系统，能排查未知问题，并回答“为什么会这样”。这对 Agent 特别重要，因为 Agent 的失败常常不是单点异常，而是 prompt、状态、工具、外部数据和模型随机性的组合结果。  
来源：[OpenTelemetry Observability Primer](https://opentelemetry.io/docs/concepts/observability-primer/)

因此，一个强 harness 不只是“有测试”，而是能回答：

- Agent 为什么这么做？
- 哪个工具返回了什么？
- 哪个状态字段被谁改了？
- 为什么进入或没有进入下一阶段？
- 成本、延迟、失败率和稳定性如何变化？
- 失败是否沉淀成了新的规则、测试或文档？

## 2. 本项目实现

### 2.1 总体判断

Travel Agent Pro 已经有比较完整的 Agent harness 雏形。它不是一个“LLM + prompt + 搜索工具”的薄封装，而是围绕旅行规划任务构建了多层控制系统：

```text
用户输入
  ↓
FastAPI / SSE
  ↓
AgentLoop
  ↓
Context / Phase / Memory / Tool Choice
  ↓
ToolEngine + 外部工具
  ↓
Guardrail / Validator / Judge / Feasibility
  ↓
Trace / Stats / Eval / Failure Analysis
  ↓
前端 Plan / Trace / Memory 可视化
```

这套系统已经覆盖了 Agent harness 的核心维度：运行时、工具边界、状态约束、评估、可观测性和项目级 Agent 文档。

### 2.2 Runtime harness：AgentLoop 和阶段状态机

核心文件：

- `backend/agent/loop.py`
- `backend/agent/hooks.py`
- `backend/agent/tool_choice.py`
- `backend/agent/reflection.py`
- `backend/phase/router.py`
- `backend/phase/prompts.py`
- `backend/phase/backtrack.py`

实现价值：

- `AgentLoop` 是 think-act-observe 主循环，控制 LLM 调用、工具调用、工具结果回灌和迭代上限。
- `max_retries` 防止 Agent 无限循环。
- `cancel_event` 和 `IterationProgress` 支持停止生成与安全继续。
- hook 机制把 validation、soft judge、phase transition gate 插入运行时。
- `ToolChoiceDecider` 强制关键阶段写入结构化状态，避免模型只输出自然语言。
- `PhaseRouter` 用状态完整度推断阶段，使 Agent 不完全依赖模型自述推进。
- `BacktrackService` 支持用户中途变更后清理下游状态。

竞争力判断：这已经是明显的 Agent runtime harness，而不是简单 chat completion wrapper。

### 2.3 Tool harness：工具注册、阶段过滤和执行边界

核心文件：

- `backend/tools/base.py`
- `backend/tools/engine.py`
- `backend/tools/update_plan_state.py`
- `backend/tools/normalizers.py`
- `backend/tools/flyai_client.py`

实现价值：

- 工具通过装饰器注册，schema 从类型信息生成。
- `ToolEngine` 负责注册、执行、错误包装和 batch 调度。
- 阶段工具白名单限制 Agent 在不同 planning phase 能调用的工具。
- `update_plan_state` 成为结构化状态写入入口，并带冗余检测和 state diff。
- `normalizers.py` 把外部 API 输出拉回统一形态，减少模型直接面对脏数据。

竞争力判断：工具不是“随便暴露给模型”，而是被阶段、schema、状态更新和 guardrail 约束。

### 2.4 Quality harness：5 层质量守护

核心文件：

- `backend/harness/guardrail.py`
- `backend/harness/validator.py`
- `backend/harness/judge.py`
- `backend/harness/feasibility.py`
- `config.yaml`

README 中已经把它定义为 5-layer harness architecture：

```text
Input → Guardrail → Agent Loop → Validator → Judge → Output
```

实现价值：

- Guardrail：注入检测、长度限制、工具输出字段校验。
- Validator：时间冲突、预算超限、天数超限、增量字段验证。
- Feasibility：在昂贵工具调用前判断明显不可行的预算/目的地/天数组合。
- Judge：用 LLM 对方案质量做软评分，并把分数接入 stats。
- Config：通过 `quality_gate`、`guardrails`、`parallel_tool_execution` 等开关控制 harness 行为。

竞争力判断：这层是项目最直接的 harness 工程亮点。它把“旅行计划是否合理”从纯模型判断拆成了规则、数据和 judge 的组合。

### 2.5 Evaluation harness：golden cases、稳定性和失败分析

核心文件：

- `backend/evals/runner.py`
- `backend/evals/stability.py`
- `backend/evals/failure_report.py`
- `backend/evals/golden_cases/*.yaml`
- `scripts/eval-stability.py`
- `scripts/failure-analysis/run_and_analyze.py`
- `scripts/failure-analysis/capture_screenshots.ts`

实现价值：

- YAML golden cases 把产品预期转化为可执行任务。
- assertions 覆盖 phase、state field、tool called、text、budget 等维度。
- `runner.py` 支持可注入 executor，便于 mock/live 双模式。
- `stability.py` 支持 pass@k 风格的多次运行评估。
- failure-analysis 脚本能执行 live 场景，采集 SSE 回复、plan state、tool calls、stats，并生成报告。

竞争力判断：项目已经具备 evaluation harness 的结构，但从现有学习文档看，eval 数据和失败闭环仍未完全产品化、持续化。

### 2.6 Observability harness：trace、stats、Jaeger 和前端 TraceViewer

核心文件：

- `backend/telemetry/stats.py`
- `backend/telemetry/setup.py`

实现价值：

- 记录 LLM call、tool call、token、duration、cost。

- memory hit 记录进入 stats 和 trace。
- OpenTelemetry + Jaeger 支持本地分布式 trace。
- 前端 TraceViewer 把 Agent 运行轨迹产品化展示。

竞争力判断：这是项目从 Demo 走向工程系统的关键。多数 Agent Demo 出问题时只能看最终回复，本项目可以追踪“模型-工具-状态-质量门控”的链路。

## 3. 本项目在 harness 工程方面做得好的和缺失的

### 3.1 做得好的地方

#### 1. Harness 不是后补测试，而是嵌入运行主路径

很多项目把 eval、guardrail、trace 当成外围工具；Travel Agent Pro 把它们接入 AgentLoop 和 hooks。比如 after_tool_call 做增量验证，before_phase_transition 做硬约束和 soft judge，ToolResult 进入 stats 和 trace。

这意味着 harness 不是“出了问题再跑”，而是在 Agent 行动过程中实时影响系统行为。

#### 2. 状态机降低了 Agent 的自由漂移

Phase 1/3/5/7 让旅行规划被拆成可观测阶段：

- Phase 1：目的地收敛；
- Phase 3：brief、candidate、skeleton、lock；
- Phase 5：日程组装；
- Phase 7：出发前清单。

这对 Agent 很重要，因为旅行规划本身是开放任务。如果没有阶段状态机，模型很容易提前订机酒、跳过约束、直接生成看似完整但不可执行的行程。

#### 3. 质量门控同时使用规则和 LLM judge

项目没有把所有判断都交给 LLM judge。预算、时间冲突、天数超限、明显不可行需求用规则拦截；开放式质量再交给 LLM judge。

这是合理分工：

- 能确定的约束，规则化；
- 需要语义判断的质量，rubric 化；
- 两者都进入 trace/stats，方便复盘。

#### 4. Eval suite 已经覆盖行为、状态和工具调用

golden cases 不是只检查最终文本，而是检查：

- 是否到达目标阶段；
- 状态字段是否写入；
- 是否调用或没有调用某个工具；
- 回复是否包含关键提醒；
- 预算是否在范围内。

这比“看回复像不像”更接近 Anthropic 建议的 Agent eval：同时检查 transcript、tool calls 和 outcome。

#### 5. 可观测性进入产品界面

TraceViewer 是一个很强的竞争力点。它把工程内部的 trace 变成产品级可视化，让人可以看到 iteration、tool call、state diff、cost、compression、validation errors、judge scores、memory hits。

这在展示中很有说服力：用户看到的不只是答案，而是 Agent 如何形成答案。

#### 6. 失败分析已经有 taxonomy

`docs/learning/2026-04-13-失败案例分析.md` 已经把失败分为：

- LLM 推理；
- 工具数据；
- 状态机；
- 约束传递；
- 设计边界。

这比“某 case failed”更有价值，因为它能指导后续修复落在哪一层：prompt、tool、state、harness 还是产品边界。

#### 7. Agent 文档体系已经开始变成 system of record

`AGENTS.md`、`PROJECT_OVERVIEW.md`、specs、plans、learning 文档组成了 Agent 可读的项目地图。尤其 `PROJECT_OVERVIEW.md` 明确要求每次 commit 同步更新，说明项目已经意识到：文档不是附属物，而是 Agent 的运行上下文资产。

### 3.2 当前缺失和风险

#### 1. Eval 还没有形成持续基线

项目有 eval runner、stability、golden cases、failure-analysis，但从现有报告看，最大的开放差距仍是：eval 数据没有稳定纳入日常开发节奏。

风险：

- 无法清楚回答“最近一次变更让 pass rate 提升还是下降”；
- 无法比较不同模型、不同 prompt、不同工具策略的效果；
- failure cases 可能停留在分析文档，而不是进入 regression suite。

建议：

- 固定一套 P0 regression eval，每次关键变更必须跑；
- 保存 `eval_reports/` 或类似目录，记录时间、commit、模型、通过率、成本、延迟；
- 把失败 case 的修复状态和对应 regression case 绑定。

#### 2. pass@k 有实现，但产品叙事还没转成稳定性指标

Anthropic 明确指出，Agent 输出有非确定性，需要多 trial 理解成功率和一致性。本项目已有 `backend/evals/stability.py`，但还需要把它提升为核心指标。

建议：

- 对每个 golden case 输出 pass@1、pass@k、pass^k 或一致性通过率；
- 对 customer-facing 场景优先看 pass^k，因为用户要的是每次都可靠；
- 在竞品/面试话术中用“稳定性曲线”替代单次 demo。

#### 3. 失败闭环还偏人工，缺少自动回填机制

当前 failure-analysis 能生成报告，但下一步应该是把失败自动转化为：

- 新 golden case；
- 新 guardrail rule；
- 新 validator；
- 新 prompt checklist；
- 新 tool normalizer；
- 新文档规则。

建议：

- 给 failure report 增加 `recommended_harness_patch` 字段；
- 每个失败必须归入一个 harness 层；
- 对“已修复”失败要求引用 regression case id。

#### 4. 架构约束还缺少更多机械化 enforcement

OpenAI 的 harness engineering 强调 custom linters、structural tests、dependency direction checks。本项目已有大量测试，但架构层 enforcement 还可以更强。

建议：

- 增加 import boundary 测试：例如 `tools` 不应依赖 `api`，`harness` 不应依赖 `frontend`，`state` 不应依赖 `llm`；
- 增加文件体积阈值或模块职责检查，避免 Agent 不断堆大文件；
- 对 prompt、tool schema、state model 做快照测试，防止无意漂移。

#### 5. 风险分类还不够系统化

项目已有 guardrail 和 memory PII 处理，但旅行场景存在更系统的风险面：

- 高龄、孕妇、儿童、残障、慢病；
- 高海拔、极端天气、签证、治安、战争/灾害；
- 金融支付、证件、联系方式、会员账号；
- 外部平台 UGC 的事实可靠性；
- 航班/酒店价格和可用性的时效性。

建议参考 NIST AI RMF 的思路，把风险纳入设计、开发、使用和评估全链路：

- 建立 travel risk taxonomy；
- 把高风险人群和目的地风险转成 validator 或 prompt checklist；
- 对 UGC 搜索结果加来源可信度和时效性标记；
- 对价格/可用性回答加 freshness 和 confidence。

#### 6. Observability 有数据，但还缺少 SLO/SLI

当前 stats 已记录 token、成本、耗时、工具调用和错误，但还缺少明确的服务目标。

建议定义 Agent SLI/SLO：

- 首 token 时间；
- 完整规划耗时；
- 每 session 成本上限；
- 工具失败率；
- phase transition 失败率；
- retry/continue 成功率；
- 每阶段平均 tool calls；
- validation error 修复成功率。

这样 trace 就不只是调试工具，而会变成工程运营指标。

#### 7. Human calibration 机制不足

LLM judge 已经存在，但 Anthropic 提醒 model-based graders 需要和人类判断校准。当前项目还缺少“人工评分样本集”和“judge 是否可信”的持续校准。

建议：

- 抽样 20-50 个 case 做人工评分；
- 记录 human score 与 judge score 的差异；
- 对 judge prompt 做版本管理；
- judge 改动时重跑 calibration suite。

#### 8. Agent-facing docs 有体系，但还可以更像导航图

`PROJECT_OVERVIEW.md` 已经很强，但 `AGENTS.md` 仍可以进一步变成目录式入口：

- 什么时候读 PROJECT_OVERVIEW；
- 修改 AgentLoop 时读哪些 docs；
- 修改 eval 时读哪些 docs；
- 修改 frontend trace 时读哪些 docs；
- 常用验证命令是什么；
- 哪些目录属于 harness，哪些属于业务。

这会进一步降低 Agent 接手任务时的搜索成本。

### 3.3 竞争力总结

如果把 Agent 项目分成 4 个层级：

| 层级 | 特征 | Travel Agent Pro 位置 |
|------|------|----------------------|
| L1 Prompt Demo | 单 prompt、多工具调用、无状态约束 | 已明显超过 |
| L2 Agent App | 有工具、有状态、有前端 | 已明显超过 |
| L3 Harnessed Agent | 有质量门控、eval、trace、失败分析 | 当前所在层级 |
| L4 Continuously Improving Agent System | eval 持续基线、自动失败回填、SLO、机械架构约束、human calibration | 下一阶段目标 |

当前项目的 harness 工程竞争力可以概括为：

> Travel Agent Pro 已经具备 L3 级 Agent harness：运行时可控、状态可验证、工具可观测、失败可分析、体验可复现。它距离 L4 的主要差距不是“没有 harness”，而是 harness 尚未完全持续化、指标化和自动闭环化。

### 3.4 下一阶段优先级

建议按以下顺序推进：

1. **建立 P0 eval baseline**  
   固定 15-25 个 regression cases，保存每次运行结果，记录 commit、模型、成本、延迟、通过率。

2. **把失败分析接入 regression loop**  
   每个已确认失败必须有 case id、root cause、harness layer、fix owner、regression status。

3. **定义 Agent SLO**  
   用现有 stats/trace 产出首 token、总耗时、成本、工具失败率、retry 成功率、phase transition 成功率。

4. **补 travel risk taxonomy**  
   把高风险人群、签证、天气、安全、海拔、支付/证件等风险变成系统化 checklist 和 eval cases。

5. **增加结构约束测试**  
   让架构边界可机械验证，减少后续 Agent 产出导致的漂移。

6. **校准 LLM judge**  
   用人工评分样本验证 judge 是否稳定，避免“评分器看起来合理但不可依赖”。

## 4. 可用于对外表达的一句话

Travel Agent Pro 的核心竞争力不是“接了几个旅行 API”，而是围绕旅行规划 Agent 构建了一套完整 harness：它能约束模型行动、验证状态正确性、观察工具链路、复现失败场景，并把失败转化为下一轮 eval 和规则。这使项目从一次性 Demo 进入了可持续工程化 Agent 的范畴。
