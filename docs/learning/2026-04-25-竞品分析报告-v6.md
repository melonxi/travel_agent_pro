### 0.3 v6 报告核心叙事

v5 报告给出的核心判断是："**端到端可展示 + 生产级容错**"。v6 在保留这一基底的同时，在三个方向上完成了"从工程到研究"的跃迁：

1. **架构跃迁**: Phase 5 引入 **Orchestrator-Workers 并行编排**（Manus pattern），由非 LLM 的 Python 编排器拆 N 个 DayTask、并行调度轻量 Day Worker、做全局校验 + 一轮带 `repair_hints` 的重派发，最终统一 `replace_all_daily_plans` 写入。这是一个在面试组合中**罕见达到"研究级"复杂度**的架构。
2. **召回跃迁**: Memory v3 + **Stage 0-4 混合召回**（硬规则 + LLM gate + 检索计划 + 三路候选 [符号/词法/语义] + reranker 证据加权）已经是工业界 RAG 标准动作，并且**真实跑通了 pass@k 报告**——这是 v5 P0 "用数据说话"的部分兑现。
3. **工程纪律跃迁**: 1,540 测试 + 7 篇复盘 + 实时约束检查 + Pending System Notes Buffer 等"细节级护城河"，证明项目不再是 demo 级原型，而是有**真实 incident 应对经验**的产品。

### 1.3 五个最有"研究级"含金量的模块

1. **`backend/agent/phase5/orchestrator.py` (823 LOC)** — Python 非 LLM 编排器，DayTask 拆分、并行调度、全局校验（POI 去重 / 预算 / 时间冲突 / 交通连接 / 节奏）、严重度分级 GlobalValidationIssue、一轮带 repair_hints 的重派发、统一 `replace_all_daily_plans` 提交。
2. **`backend/agent/phase5/day_worker.py` (368 LOC)** — 轻量 LLM Worker，4 重收敛护栏：去重查询抑制 / 恢复链预算 / 末轮锁死 / JSON 修复预算；错误码 REPEATED_QUERY_LOOP / RECOVERY_CHAIN_EXHAUSTED / JSON_EMIT_FAILED / TIMEOUT / LLM_ERROR / NEEDS_PHASE3_REPLAN。
3. **`backend/memory/recall_reranker.py` (897 LOC)** — 证据加权 reranker：lane_fused 0.25 / semantic 0.15 / lexical 0.08，source-aware min-max 归一化，source priors，避免长尾抖动。
4. **`backend/memory/recall_gate.py` + `recall_signals.py`** — 三层准入闸：Layer 1 信号抽取（6 类词）/ Layer 2 P1-P6 规则引擎 / Layer 3 LLM gate；最大限度避免误召。
5. **`backend/phase/prompts.py` (871 LOC)** — 10 段技能卡架构（角色/目标/硬法则/回复原则/阶段边界/工具契约/状态写入契约/完成 Gate/Red Flags/压力场景）+ `build_phase3_prompt(step)` 动态分步装配（brief/candidate/skeleton/lock）。

### 1.4 工程细节级护城河（v6 新增）

- **Pending System Notes Buffer**: 并行 tool_calls 的 SYSTEM 消息（如 `[实时约束检查]`）缓存到 session-level buffer，下一次 LLM 调用前 flush，保证 `assistant.tool_calls → all tool replies` 的原子性。
- **实时约束检查**: 工具调用中检测约束违反，作为 SYSTEM 消息注入下一轮，让模型自我修复而不是事后回滚。
- **`classify_opaque_api_error`**: 处理讯飞类网关返回的裸 APIError，准确分类 TRANSIENT / RATE_LIMITED / BAD_REQUEST，避免错误降级。
- **KV-Cache 优化**: Phase 5 Workers 共享 system prompt 前缀，估测命中率约 93.75%。
- **POI 全局唯一性**: locked_pois / candidate_pois 拥有边界硬约束，语义 POI 重复校验器（不止字符串 match）。
- **NEEDS_PHASE3_REPLAN 协议**: Worker 在结构性不可能时主动发信号，触发 Phase 3 回退而非死循环。

## 4. 叙事力评估（面试可讲的故事数量）

### 4.1 v5 已有的 7 条主线（保留）

1. 端到端可展示
2. LLM 容错三层架构（前置/熔断/工具级）
3. Phase Router + State Machine
4. 工具引擎 + 严格 schema
5. SSE 流式 + 前端 keepalive
6. 多 LLM Provider 抽象
7. Memory v2 + 极简 KV

### 4.2 v6 新增的 8 条主线（重磅）

8. **Phase 5 Orchestrator-Workers 并行编排**（Manus 模式）—— "为什么不用 LangGraph？因为 LangGraph 的状态机做并行需要写很多 lambda 节点；我们的场景是 N 天独立子任务，Manus 的 Plan→Workers→Aggregate 模式更直接，KV-Cache 命中率约 93.75%"
9. **Worker 4 重收敛护栏** —— "Worker 怎么防死循环？去重查询抑制、恢复链预算、末轮锁死、JSON 修复预算；外加 NEEDS_PHASE3_REPLAN 信号让 Worker 主动跳出不可能场景"
10. **Stage 0-4 混合召回管道** —— "Stage 0 硬规则、Stage 1 LLM gate、Stage 2 检索计划、Stage 3 三路候选（符号/词法/语义）、Stage 4 reranker 证据加权"
11. **三层 Recall Gate** —— "为什么不直接向量检索？因为旅行域有大量噪声，三层 gate 能在 80% 噪声 query 上提前剪枝，pass@k=0.90 就是这么得来的"
12. **Reranker 证据加权** —— "lane_fused 0.25 / semantic 0.15 / lexical 0.08，source-aware min-max 归一化 + source priors，避免长尾分数抖动"
13. **Skill-Card Prompt 架构** —— "10 段固定结构：角色/目标/硬法则/回复原则/阶段边界/工具契约/状态写入契约/完成 Gate/Red Flags/压力场景；可以做 prompt diff、可以局部更新"
14. **Plan Tools 拆分 + 共享 plan_writers** —— "为什么不留一个 update_plan_state？因为 LLM 误用大工具的概率远高于小工具；6 个单职责工具 + 共享纯 mutation 层，错误率显著下降"
15. **Pending System Notes Buffer** —— "并行 tool_calls 的 SYSTEM 消息（实时约束检查）必须保证 assistant.tool_calls → all tool replies 原子性，所以缓存到 session buffer，下一次 LLM 调用前 flush"

### 4.3 叙事密度评分

v5: 7 条主线 → 评分 8.5 / 10
v6: **15 条主线** → 评分 **9.5 / 10**（每个主线都有可量化的工程决策点）

## 9. 面试话术（v6 高杠杆 Q&A 库）

### Q1: 为什么 Phase 5 用并行 Orchestrator-Workers，不用 LangGraph？

> "LangGraph 的 state machine 做并行需要写很多 lambda 节点，状态合并要自己处理；我们 Phase 5 的本质是 N 天独立子任务 + 一次全局校验。Manus 论文的 Plan→Workers→Aggregate 模式更直接：Python 编排器（非 LLM）做拆分和校验、Worker 做生成、最后统一提交。这样每个 Worker 共享 system prompt 前缀，KV-Cache 命中率约 93.75%；而且非 LLM 编排器可以用普通单测覆盖，不像 state machine 那样要 mock LLM。"

### Q2: Worker 怎么防止死循环？

> "4 重收敛护栏：第一，去重查询抑制——同一 query 重复出现就拒；第二，恢复链预算——一个 Worker 最多走 K 跳工具修复链；第三，末轮锁死——最后一轮强制只允许 emit 不允许 query；第四，JSON 修复预算——LLM 输出畸形 JSON 时最多修 N 次。再加上 NEEDS_PHASE3_REPLAN 这个特殊错误码：Worker 自检发现结构性不可能（比如 skeleton POI 数量不够）就主动跳出，触发 Phase 3 回退而不是硬撞。"

### Q3: 召回怎么避免误召？

> "三层 Recall Gate。Layer 1 信号抽取：把 query 分成 6 类词（POI/时间/预算/约束/偏好/否定）。Layer 2 P1-P6 规则引擎：硬规则剪枝，比如 query 里没有任何旅行域信号就直接跳过。Layer 3 才轮到 LLM gate 做语义判断。三层组合下，80% 的噪声 query 在 LLM 调用前被剪掉，pass@k=0.90 的可靠性就来自这个早剪枝。"

### Q4: Reranker 的 evidence 权重怎么设计？

> "三个证据来源加权：lane_fused 0.25（三路融合分数）、semantic 0.15（语义独立分数）、lexical 0.08（词法独立分数）。lane_fused 权重最高是因为它已经是综合信号；single-lane 分数权重低是为了避免某条 lane 的 outlier 主导排名。每条 lane 单独做 source-aware min-max 归一化，避免不同来源的分数尺度不一致；最后叠加 source priors（episode > slice > profile），保证更具体的证据排前。"

### Q5: 为什么把 update_plan_state 拆成 6 个工具？

> "因为 LLM 误用大工具的概率远高于小工具。原来的 update_plan_state 接受 dict-of-anything，LLM 经常把 must_do 写到 trip_brief.must_do 而不是 constraints。拆成 set_trip_brief / add_constraints / add_preferences / replace_all_daily_plans / 等 6 个单职责工具后，每个工具的 schema 极窄，LLM 错的空间被压缩。底层共享 state/plan_writers.py 纯 mutation 层（327 行），单测好写。"

### Q6: Pending System Notes Buffer 解决什么问题？

> "OpenAI tool_calling 协议要求一个 assistant.tool_calls 后必须紧跟所有 tool reply，中间不能插 SYSTEM 消息。但我们有'实时约束检查'——工具执行中检测到约束违反时要注入 SYSTEM 消息让模型自我修复。直接 append 会破坏原子性。所以缓存到 session-level pending_system_notes buffer，下一次 LLM 调用前再 flush。这是个细节但是非常容易被忽略，错了会导致 OpenAI 端 400 错误。"

### Q7: 真实 pass@k=0.90 这个数据怎么来的？

> "`docs/recall-reranker-eval.md` 里跑的：6 个 recall golden case，每个 case 跑 5 次（k=5），共 30 次评测；统计每个 case 的 pass 比例，平均得 0.90。其中 recall-003 是 2/5（不稳定），其他基本是 5/5 或 4/5。报告里有 cost / latency 数据。要诚实说：这只是 6 个召回 case，端到端 34 case 还没全跑——这是 v6 → v7 的 P0。"

### Q8: 11 天独立写 38k 行，质量怎么保证？

> "1,540 测试是底线，TDD 几乎每个新模块。第二是 7 篇真实 postmortem——比如 phase5-promise-only-loop-termination、phase5-parallel-guard-refactor，每次出问题都写复盘，避免下次撞同一个坑。第三是 PROJECT_OVERVIEW.md 强制每次 commit 同步更新，保证架构文档不漂移。第四是工具拆分 + 共享 plan_writers，让每个改动的爆炸半径可控。"

### Q9: 为什么不用 Cursor / Cline 这种现成 Agent Framework？

> "因为旅行规划这个域的特殊性：它有强 phase 边界（认识用户 → 候选 → skeleton → lock → 生成）、强状态写入约束（POI 全局唯一、时间不冲突）、和强失败模式（不可行的预算/天数）。现成 Framework 的通用性反而是负担——它们的 state 都是 dict-of-anything，没法做 phase 级硬法则。我们的 Phase Router + Skill-Card Prompt + 单职责工具是为这个域定制的，可控性远高于通用 Framework。"

### Q10: 如果让你再做一次，会改什么？

> "三件事。第一，从 day 1 开始就跑 Eval CI，不要等到第二阶段才补；这次我有 34 case 但没有 CI，导致重构时心理负担大。第二，更早做 prompt diff 工具——我们的 prompt 变更经常出现回归，但缺手段定位是哪段 prompt 改坏了。第三，更早把 LLM Provider 抽象做成 supplier rotator，不要等讯飞挂了才补 classify_opaque_api_error。"
