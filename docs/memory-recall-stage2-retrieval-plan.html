Stage 2：Retrieval Plan

Stage 2 把 Stage 1 的“需要召回”翻译成可执行的结构化查询计划。它不直接取记忆，而是决定：查哪类来源、哪些领域、哪个目的地、用哪些关键词、取多少条。

核心职责

它回答的问题是：去哪里查，怎么查。

Query Contract

结构化 schema 是 Stage 2 的核心产物。
字段
允许值 / 规则
为什么重要
source
profile / episode_slice / hybrid_history
区分查长期画像、历史切片，还是两者都查。
buckets
仅 profile / hybrid_history 必填。
限制查 constraints、rejections、stable_preferences、preference_hypotheses。
domains
只能用系统枚举：food、hotel、pace、flight、budget 等。
防止模型造开放字段，保证后续可解析。
destination
目的地字符串，可为空。
用于 destination filter / normalization，避免错查其他城市。
keywords
关键词列表。
给 Stage 3 的 symbolic、lexical、semantic lane 提供检索方向。
top_k
正整数，最多 10。
控制召回预算，避免候选过多。
reason
160 字以内。
可观测性：解释为什么这样查。

输入上下文

Stage 2 的输入比 Stage 1 更“检索化”。

作用边界

Stage 1 输出。

完整 Prompt

当前真实发送给 Retrieval Plan LLM 的模板。
你是旅行记忆召回规划器。
Recall gate 已经确认需要召回。你不要重新判断要不要 recall，只输出检索计划。
检索计划必须主要基于 latest_user_message。
previous_user_messages 只用于解析 latest_user_message 中的省略、指代和承接关系，不能让更早消息覆盖当前请求。
plan_facts 只用于抽取目的地、当前对象、预算、同行人等检索参数，不用于重新判断是否需要 recall。
你的计划会驱动符号检索，不是语义向量检索；domains、destination、keywords 必须精确、可被下游字符串匹配消费。
source 决策规则：profile_preference_recall -> profile；profile_constraint_recall -> profile；past_trip_experience_recall -> episode_slice 或 hybrid_history；mixed_or_ambiguous -> hybrid_history。
buckets 仅在 source=profile 或 hybrid_history 时填写。profile_constraint_recall 优先 constraints/rejections；profile_preference_recall 优先 constraints/rejections/stable_preferences；mixed_or_ambiguous 可加 preference_hypotheses。
合法 domains 只有：["itinerary","pace","food","hotel","accommodation","flight","train","budget","family","accessibility","planning_style","documents","general"]
destination 只允许填写目的地名称；无法可靠推断时填空字符串。
top_k 表示每个 source 的候选预算，不是总候选数。
reason 只写一行简短遥测说明，不要展开推理过程。
latest_user_message={latest_user_message}
previous_user_messages={previous_user_messages}
gate_intent_type={gate_intent_type}
gate_reason={gate_reason}
gate_confidence={gate_confidence}
stage0_signals={stage0_signals}
plan_facts={plan_facts}
memory_summary={memory_summary}
示例1：{"source":"episode_slice","domains":["hotel"],"destination":"大阪","keywords":["住宿","酒店"],"top_k":3,"reason":"past_trip_experience_recall -> Osaka hotel slices"}
示例2：{"source":"profile","buckets":["constraints","rejections","stable_preferences"],"domains":["hotel"],"destination":"","keywords":["住宿","偏好"],"top_k":3,"reason":"profile_preference_recall -> hotel preference profile"}
必须调用 build_recall_retrieval_plan 工具输出结果。

这里保留“必须调用工具”的提示，是因为 Stage 2 的核心产物就是结构化 plan；代码层也会强制工具调用，prompt 层这句话主要是在语义上防止模型解释或重新做 gate 判断。

工具 Schema

oneOf 让不同 source 有不同字段约束。
{

  "description": "Generate a retrieval plan for symbolic travel-memory lookup after the recall gate has already decided that recall is needed. Choose the smallest plan that can drive exact domain/keyword/destination matching.",
  "parameters": {
    "type": "object",
    "oneOf": [

        "additionalProperties": false
      },
      {
        "source": "hybrid_history",
        "required": ["source", "buckets", "domains", "destination", "keywords", "top_k", "reason"],
        "additionalProperties": false
      }
    ]
  }
}
CONTRACT

为什么用 oneOf

不同 source 需要不同字段。episode_slice 没有 profile bucket；profile 和 hybrid_history 必须指定 bucket。oneOf 能让错误结构尽早失败。

为什么 additionalProperties=false

Stage 3 只消费固定字段。禁止额外字段可以避免模型发明 filter、score、semantic_query 之类下游不认识的参数。

intent / 场景
推荐 source

profile_constraint_recall

Domain 和 Bucket 枚举

这些词是可执行检索合同，不是自然语言标签。
DOMAINS

允许的 domains

  • itinerary：行程、日程、路线。
  • pace：节奏、体力、是否轻松。
  • food：饮食、餐厅、忌口。
  • hotel / accommodation：酒店、住宿、民宿、青旅。
  • flight / train：航班、火车、高铁。
  • budget：预算、价格敏感度。
  • family / accessibility：同行人、老人儿童、无障碍。
  • planning_style / documents / general：规划风格、证件材料、兜底。
BUCKETS

允许的 profile buckets

  • constraints：稳定硬约束，例如不能坐红眼。
  • rejections：明确拒绝和避雷，例如不吃鱼、不住青旅。
  • stable_preferences：稳定偏好，例如喜欢自然味道、轻松节奏。
  • preference_hypotheses：弱假设，只在 mixed/ambiguous 时谨慎加入。

为什么不直接让 Stage 3 自己猜

分离“语义判断”和“检索参数”。
Precision

查错来源比查不到更危险

住宿偏好应该查 profile，过去踩坑可能查 episode_slice。计划先定 source，能减少无关历史进入候选池。

Schema

约束模型输出

source-aware schema 明确哪些字段允许出现。无效 payload 会回退到 fallback plan，不让错误结构继续传染。

Travel

旅行领域需要目的地语境

“大阪住宿”和“东京住宿”的历史经验不一定可迁移。destination 字段让后续过滤和归一化有依据。

如何影响 Stage 3 / Stage 4

Plan 不是展示字段，它直接改变候选池。
Stage 3

控制候选来源

source=profile 只查画像；episode_slice 只查历史切片；hybrid_history 两路都查。

Stage 3

控制匹配方向

domains、destination、keywords 会影响 symbolic ranking、destination match 和 keyword overlap。

Stage 4

影响 reranker 意图

Reranker 会读取 retrieval plan 的 source、domains、keywords 和 reason，决定哪些候选更贴近当前请求。

失败回退

Query Tool 超时或异常时走启发式计划。
Timeout

query_plan_timeout

按用户消息和 Stage 0 signals 生成 heuristic retrieval plan。

Error

query_plan_error

模型调用失败时仍保留可执行 plan，避免整轮召回中断。

Invalid

invalid_query_plan

解析失败时使用默认 hybrid fallback，保留 telemetry 便于追踪。

Stage 2 有效的关键在于：它让召回从“自然语言愿望”变成“稳定、可验证、可回退的检索合同”。
