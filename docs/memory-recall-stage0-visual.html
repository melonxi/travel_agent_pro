记忆召回 Stage 0：硬规则短路

Stage 0 是记忆召回链路的第一道闸门。它不调用 LLM，只看最后一条用户消息中的固定词表信号，然后快速输出三类结果： 直接召回、直接跳过、或交给 LLM gate。

代码来源

它在完整召回链路中的位置

Stage 0 只负责第一步，后面还有 LLM gate、检索计划、候选生成和重排。
Stage 0
硬规则短路
词表匹配 + 优先级规则，输出三类决策。
Stage 1
LLM Gate
只处理 undecided，判断是否需要召回。
Stage 2
Retrieval Plan
决定查 profile、slice 还是 hybrid_history。
Stage 3
候选生成
symbolic / semantic / lexical lane 产出候选。
Stage 4

Stage 0 的核心流程

按优先级从上到下判断；第一个命中的规则直接返回。

三类输出怎么理解

这是理解 Stage 0 的关键：规则很多，出口只有三个。
DIRECT
force_recall

规则层已经足够确定：用户在引用历史、习惯或偏好。直接进入后续召回，不再问 LLM gate 要不要召回。

STOP
skip_recall

规则层已经足够确定：用户只是问当前行程事实，或只是确认/继续。不要查历史记忆，避免噪音。

DEFER
undecided

规则层不做最终判断。把最近用户消息窗口交给 LLM gate，让模型结合上下文判断是否需要召回。

为什么要有 Stage 0

Know why：它不是为了聪明，而是为了把明确样本从不稳定模型判断里拿出来。
WHY

它解决的是召回门控的可靠性问题

记忆召回不是越多越好。召回太少会漏掉用户长期偏好，召回太多会把无关历史塞进 prompt。 Stage 0 用硬规则先处理最明确的两端：明显要召回的历史/偏好线索，以及明显不该召回的当前事实和确认语。

DESIGN

它把判断拆成“确定”和“不确定”

确定要召回
例如“按我的习惯”“像上次”，规则直接输出 force_recall，减少 LLM gate 漏判。
确定不用召回
例如“这次预算多少”“继续”，规则直接输出 skip_recall，避免污染上下文。
不确定
例如推荐、兜底、否定旧偏好，统一输出 undecided，交给 Stage 1 用上下文判断。

Stage 0 的核心设计不是“尽量自己判断”，而是“只在高确定性场景抢先判断”。这让它可解释、可测试，也不和 LLM gate 抢复杂语义理解的职责。

为什么这些场景不直接在 Stage 0 决策

Know why：很多用户话术看起来像关键词，其实需要上下文。

流程为什么按这个顺序设计

Know why：优先级不是随便排的，它在控制漏召回和误召回的边界。
1

先处理空消息

空消息没有语义，直接进入 undecided。这样不会把空输入误判成确认、事实问题或推荐请求。

2

历史/偏好信号优先

“上次”“按我的习惯”是最强召回信号。它们放在前面，能避免被“这次酒店”等事实词抢先拦截。

3

推荐信号早于事实跳过

“这次酒店怎么安排比较好”同时含有事实词和推荐词。先看 recommend，才能把个性化推荐交给 LLM gate。

4

事实和确认才直接跳过

只有非常像当前计划查询或确认操作时才 skip_recall，降低把用户偏好请求误当成普通事实问句的风险。

这个顺序有效，是因为它把“强召回信号”放在最高优先级，把“强跳过信号”限制在更窄条件里，并把所有有语义歧义的样本交给 Stage 1。工程目标不是让 Stage 0 覆盖所有语言，而是让它稳定处理高置信度样本。

开发者改规则时的 Know-how

Know how：先判断问题属于“硬信号缺失”还是“语义判断缺失”。
WHEN

什么时候改 Stage 0

  1. 某个话术总是高确定性，例如“照旧”“按我的习惯”。
  2. 它不需要上下文也能稳定判断为召回或跳过。
  3. 加入词表后不会明显误伤其他意图。
  4. 能写成小而明确的规则测试。
WHEN

什么时候改 Stage 1 prompt

  1. 用户话术依赖主观语义，例如“好吃”“舒服”“适合老人”。
  2. 需要结合最近几轮消息或记忆概览才能判断。
  3. 同一个词在不同上下文下可能应该召回，也可能不该召回。
  4. 目标是提升模型对“个性化请求”的理解边界。

实用判断：如果你能用“只要出现这个短语就一定这样处理”描述它，优先改 Stage 0；如果你必须说“要看上下文”，优先改 Stage 1。

设计取舍

为什么 Stage 0 不直接做成智能判断器。
FAST

便宜、稳定、可回归

Stage 0 是纯字符串规则，不消耗 LLM，不受模型波动影响。适合处理明确的强信号和明确的跳过场景。

SAFE

不确定就交给模型

推荐类、否定旧偏好、兜底样本都输出 undecided，让 Stage 1 用最近消息窗口和记忆概览做更细判断。

LIMIT

只看最后一句

Stage 0 当前只处理最后一条 user message。承接语、多轮上下文依赖主要由 Stage 1 和 Stage 2 补足。

记住一句话：Stage 0 不是“完整理解用户意图”的地方，而是“快速处理显而易见的情况”的硬规则闸门。
