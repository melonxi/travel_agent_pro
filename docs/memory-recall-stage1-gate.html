Stage 1：LLM Gate

Stage 1 只处理 Stage 0 输出 undecided 的样本。它用真实语言理解判断：当前请求是否需要调取用户画像、历史经验或工作记忆。

输入和输出

它回答的问题是：要不要召回？

并不是每一轮都会调用 LLM gate。

Gate Window 和当前事实

Stage 1 看压缩后的语义上下文，不看记忆库存。
WINDOW

最近用户窗口

  • 最多取最近 3 条非空用户消息。
  • 从最近一条消息开始，向前回填历史消息。
  • 最近一条用户消息必须完整保留，即使超过 1200 字符。
  • 1200 字符预算只约束更早的用户消息。
  • 更早消息只会整条加入或整条丢弃，不做半截裁剪。
  • 传入 prompt 时拆成 latest_user_message 和 previous_user_messages。
  • 用途：以最近一条为主判断，前文只辅助理解“换个吧”“再轻松点”这类承接语。
FACTS

当前旅行事实

  • 来自当前 TravelPlanState 快照。
  • 用于识别“这次预算是多少”这类当前事实问题。
  • 不包含 profile count、profile key 或 working memory preview。
  • 不告诉模型当前有没有可召回记忆。

这里刻意不传任何记忆库存概览。Stage 1 只判断“这句话语义上是否需要召回”，不能把“当前有没有记忆库存”混进信号判断。即使记忆库为空，“想去有好吃的地方玩”这类请求也应该被判定为需要尝试召回，后续阶段再处理 zero hit。

完整 Gate Prompt

这是当前真实发送给 LLM gate 的提示词模板。
你是旅行记忆召回判定器。
你的任务是判断：为了更好地回答用户当前请求，是否应该调取用户的长期画像、历史偏好、长期约束、过去旅行经验或当前会话工作记忆。
判定对象：必须以 latest_user_message 为主要判断依据。
previous_user_messages 只用于理解 latest_user_message 中的省略、指代和承接关系，不能因为更早消息本身需要召回，就把当前轮判成 needs_recall=true。
current_trip_facts 只用于识别 latest_user_message 是否在询问当前行程中已经明确存在的事实。
如果 latest_user_message 是对当前行程做个性化评价、选择、优化、推荐或取舍，即使它引用了 current_trip_facts，也应倾向于 needs_recall=true。
核心原则：
1. 如果用户请求涉及个性化选择、推荐、排序、取舍、规划风格、目的地选择、住宿选择、交通选择、餐饮偏好、活动偏好、预算倾向、体力节奏、同行人需求、避雷避坑，即使用户没有明确说“按我的偏好”或“像以前一样”，也应倾向于 needs_recall=true。
2. 如果用户表达的是“我想要/我喜欢/我不喜欢/我偏好/我受不了/我希望/我在意/我不想要”这类个人特征、偏好或约束，并且这些信息可能影响后续推荐或规划，应倾向于 needs_recall=true，用于结合已有画像避免重复提问或冲突建议。
3. 如果用户提出开放式旅行请求，例如“想去好玩的地方”“想去好吃的地方”“帮我安排住宿”“这几个哪个更适合”“怎么安排比较好”，这通常需要个性化判断，应召回相关画像和历史经验。
4. 如果用户只是在询问当前行程中已经明确存在的事实，例如“我这次预算是多少”“酒店是哪家”“几号出发”“现在目的地是哪里”，不需要召回历史记忆，needs_recall=false。
5. 如果用户只是确认、继续、寒暄、系统操作或没有实际规划含义，例如“好的”“继续”“就这个”“重新开始”，通常 needs_recall=false。
6. 如果当前请求既可能是普通查询，也可能受个人偏好影响，采用保守召回策略，needs_recall=true。召回比漏召更可接受，因为后续 reranker 会过滤无关记忆。
请特别注意：
- 不要只在用户显式提到“历史、上次、以前、按我偏好”时才召回。
- 用户说“好吃、舒服、轻松、不折腾、适合我、好玩、自然、安静、方便、省心、亲子、老人友好、预算别太高”等主观标准时，通常都需要召回用户画像。
- 如果当前请求需要回答“什么更适合这个用户”，就应该召回。
- 如果只是回答“当前计划里已经写了什么事实”，才跳过召回。
latest_user_message={latest_user_message}
previous_user_messages={previous_user_messages}
current_trip_facts={current_trip_facts}
输出字段语义：needs_recall 表示 latest_user_message 是否需要召回；intent_type 必须使用工具枚举；reason 简短说明为什么需要或不需要召回；confidence 为 0 到 1 的置信度。

这个 prompt 的核心不是“看到历史词才召回”，而是把旅行中的主观选择、推荐、避坑、节奏、预算、同行人需求都视为可能需要个性化记忆的场景。

WHY TOOL

为什么强制工具输出

自然语言回答不稳定，后续代码需要确定字段。工具 schema 把模型输出限制成结构化 JSON，并让 parser 能检查 intent 和 needs_recall 是否自洽。

为什么 intent_type 要枚举

召回 gate 不只是 true/false。intent 能告诉后续 trace 和 fallback：这是当前事实、画像偏好、画像约束、历史经验，还是混合模糊场景。

出参和解析规则

parser 会拒绝自相矛盾或无效 payload。

true 进入 Stage 2；false 设置 recall_skip_source=gate_false。
intent_type
必须在枚举内。true 只能配召回类 intent；false 只能配非召回类 intent。

为什么需要 LLM Gate

Stage 0 不猜复杂语义，Stage 1 才做语义判断。
Language

主观请求不一定有关键词

“想去好吃的地方玩”没有命中 Stage 0，但旅行语义上需要饮食画像。LLM gate 负责识别这类个性化意图。

Context

主次分明地看窗口

latest_user_message 是判定对象；previous_user_messages 只用于省略、指代、承接消歧，不能单独触发召回。

Boundary

不看记忆库存

Gate 的职责是判断语义需求，不判断当前库里有没有命中项。库存、候选数量和 zero hit 交给后续阶段处理。

为什么这样设计有效

把复杂语义留给模型，把结构约束留给代码。
Recall

降低漏召回

旅行推荐里很多需求没有显式“按我偏好”，但仍强依赖画像。Prompt 把“什么更适合这个用户”定义为召回场景。

Separation

避免库存污染判断

不把记忆库存概览放进 prompt，模型就不会因为“看起来记忆少”而把本应召回的个性化请求判成 false。

Robust

模型失败可回退

timeout、error、invalid payload 都不会让链路崩掉，而是按 Stage 0 signals 走启发式 fallback。

失败和保守策略

模型不稳定时不能让链路无声失败。
Mixed

mixed_or_ambiguous

即使模型返回 false，也会归一化为召回，避免模糊样本漏召。

Failure

timeout / error

进入 stage0-aware heuristic fallback。若 Stage 0 有历史/推荐线索，尽量保守召回。

Risk

当前风险

真实模型调用受 timeout 影响。若 timeout 太短，P6 个性化请求可能 fallback 为不召回。

Stage 1 的有效性来自分工：Stage 0 处理高确定规则，Stage 1 处理语言语义，后续 Stage 4 再过滤无关记忆。因此“保守召回”通常比“漏召回”更可接受。
