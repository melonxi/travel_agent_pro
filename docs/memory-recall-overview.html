记忆召回机制总览

这套召回架构把“是否需要记忆、该查哪里、怎么拿候选、最终选哪些”拆成 5 个 stage。它服务的是旅行规划场景：用户偏好会强烈影响方案质量，但当前行程事实又必须保持权威、干净、可解释。

入口文件
backend/main.py
backend/memory/manager.py
backend/memory/recall_*.py

各 Stage 入口

每页都解释 what / how / why / travel fit。
Stage 0

硬规则短路

用词表和优先级规则先处理强确定样本：直接召回、直接跳过或交给 LLM gate。

 Stage 1

LLM Gate

对 undecided 样本做语义判断：当前请求是否需要长期画像、历史经验或工作记忆。

 Stage 2

Retrieval Plan

把“需要召回”转成结构化查询计划：source、buckets、domains、destination、keywords。

 Stage 3

候选生成

按查询计划从 profile 和 episode slices 中生成候选，并附带 lane evidence。

 Stage 4

Reranker

融合规则分、source prior、语义/词法证据，选出最终进入 prompt 的记忆。

整体数据流

每一层只回答一个问题。

为什么要这样设计

核心是旅行规划里的“个性化”和“当前事实权威”之间的张力。
Problem

旅行偏好强个性化

同一句“想去好吃的地方玩”，不同用户的答案可能完全不同：有人不吃辣、有人带孩子、有人讨厌赶路。没有历史画像，Agent 容易给出泛泛推荐。

Problem

当前行程事实不能被历史污染

“这次预算多少”“酒店是哪家”应该从 TravelPlanState 取权威事实，而不是从旧旅行记忆里猜。否则会把过去计划误带入当前计划。

Architecture

先判定，再检索，再重排

如果把所有历史都常驻 prompt，会增加噪音和成本；如果只靠单次向量检索，会缺少规则可解释性。分层后，每一步都能单独观察和回归。

UX

召回结果要可解释

前端 trace 可以看到 gate、query plan、candidate_count、reranker scores。用户和开发者能判断：是没召回、没命中，还是被 reranker 过滤了。

和旅行场景的适配

不是通用聊天记忆，而是服务行程规划的记忆。
Profile

长期画像

饮食禁忌、住宿偏好、预算倾向、体力节奏、亲子/老人需求。适合跨旅行复用。

Working

当前工作记忆

当前 session/trip 内的临时拒绝、局部取舍和短期上下文。只服务当前会话。

Slices

历史旅行切片

过去行程模式、住宿选择、交通选择、踩坑和被拒选项。适合从经验中迁移。

当前旅行事实仍由 TravelPlanState 权威提供；记忆召回只负责补充“这个用户可能更适合什么”。
