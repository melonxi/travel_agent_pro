Stage 3：候选生成

Stage 3 真正从记忆存储中拿候选。它接收 Stage 2 的查询计划，构造 Query Envelope，按 source policy 搜索 profile 和 episode slices，并返回候选与 evidence sidecar。

从 Plan 到 Candidates

先规范化查询，再走多条检索 lane。

三条 Lane

当前生产默认 symbolic + semantic；lexical 在 feature flag 后。
Lane
怎么找
适合什么旅行记忆
symbolic

明确标签的偏好：不吃辣、避开红眼、喜欢轻松节奏。

semantic

语义相近但词不完全一样的历史经验或偏好。

Stage 3 不再问“要不要召回”，也不再让 LLM 判断。它的职责是把 Stage 2 的 plan 执行成一批带证据的候选，后续由 Stage 4 做精排和最终过滤。

心智模型

把 Stage 3 想成一个“证据收集器”。
1

先确定搜索边界

Stage 2 已经给出 plan。Stage 3 先把 plan 变成 source policy：到底查 profile、episode slice，还是两者都查。

2

再扩展可匹配表达

把 domains、keywords、destination 做扩展和归一化，让“住宿/酒店/民宿”“大阪/关西”这类表达能被同一套 lane 消费。

3

多路找候选并合并证据

Symbolic 找结构化强证据，Lexical 找词面重叠，Semantic 找语义相似；同一候选多路命中时合并 evidence。

一句话模型：Stage 3 不负责最终判断“这条记忆该不该用”，它负责“尽可能把可能相关的记忆拿出来，并说明为什么拿出来”。

完整流程图

从 plan 到 fused candidates 的执行顺序。

伪代码

开发者可以按这个顺序读源码。
def retrieve_recall_candidates(plan, profile, slices, user_message, travel_plan, config):
    envelope = build_query_envelope(
        query=plan,
        user_message=user_message,
        plan=travel_plan,
        config=config,
    )

    telemetry = init_stage3_telemetry(envelope, fallback_used=plan.fallback_used)
    lane_results = []

    if config.symbolic.enabled:
        result = SymbolicLane().run(envelope, profile, slices, config)
        record_lane_result(telemetry, result)
        lane_results.append(result) if not result.error else None

    if config.lexical.enabled:
        result = LexicalLane().run(envelope, profile, slices, config)
        record_lane_result(telemetry, result)
        lane_results.append(result) if not result.error else None

    if config.semantic.enabled:
        result = SemanticLane().run(
            envelope,
            profile,
            slices,
            config,
            embedding_provider,
        )
        record_lane_result(telemetry, result)
        lane_results.append(result) if not result.error else None

    fused = fuse_lane_results(lane_results, config.fusion)
    telemetry.total_candidates_before_fusion = sum(len(r.candidates) for r in lane_results)
    telemetry.total_candidates_after_fusion = len(fused)
    telemetry.zero_hit = len(fused) == 0

    return Stage3RecallResult(
        candidates=[item.candidate for item in fused],
        evidence_by_id={item.candidate.item_id: item.evidence for item in fused},
        telemetry=telemetry,
    )

Query Envelope

在跑 lane 之前先把 query 规范化。
Source Policy

把 source 变成布尔开关

profile 表示只查画像，episode_slice 只查历史切片，hybrid_history 两边都查。lane 不需要重复理解 source 字符串。

Expansion

扩展 domain / keyword

hotel 会扩到 accommodation，住宿 会扩到酒店、民宿、住哪里。这样 Stage 2 不必穷尽所有同义词。

Destination

目的地归一化

启用后会计算 canonical、aliases、children、region，让“大阪 / 关西 / 京都周边”这类地理关系可被后续 lane 和 telemetry 解释。

Symbolic Lane

规则和结构化字段优先，适合高置信记忆。
HOW

它怎么做

  • 根据 source policy 决定查 profile、slice，或两者都查。
  • profile 侧调用 rank_profile_items()，只遍历 plan 指定 buckets。
  • slice 侧调用 rank_episode_slices()，使用 destination、domain、keyword 等规则排序。
  • 每侧最多取 min(symbolic.top_k, plan.top_k)。
  • 输出 RetrievalEvidence：lanes、matched_domains、matched_keywords、destination_match_type、retrieval_reason。
WHY

为什么需要它

用户画像里的约束和偏好通常是结构化的，例如“不吃辣”“不坐红眼”“住宿不要青旅”。这类记忆不需要向量相似度，规则匹配更可解释、更稳定，也更容易做错误定位。

主要取舍

优点是确定性强、证据清晰；缺点是依赖 Stage 2 的 domains / keywords / destination 质量，词面没覆盖时可能漏召。

Lexical Lane

词面重叠匹配，当前默认关闭，可按配置打开。
HOW

它怎么做

  • 把 user_message、expanded domains、expanded keywords 拼成 query text。
  • tokenizer 同时支持 ASCII token、中文连续串、单字和 bigram。
  • profile 文本来自 value、applicability、context、recall_hints、source_refs。
  • slice 文本来自 entities、keywords、content、applicability。
  • 用 query terms 和 document terms 的 F1 overlap 打分，低于 0.05 过滤。
  • 按分数降序取 lexical.top_k。
WHY

为什么需要它

Lexical lane 是 symbolic 和 semantic 之间的中间层。它不依赖向量模型，也不像 symbolic 那样强依赖手写规则。对于“酒店 / 住宿 / 民宿”“吃 / 餐厅 / 美食”这类明确词面相关但字段不完美的记忆，它能补召。

为什么默认关闭

词面 overlap 容易被短词和高频词污染，特别是中文单字 token。当前实现保留 feature flag，是为了在观测 precision 后再决定是否长期打开。

Semantic Lane

向量相似度兜住“意思相近但词不同”的情况。
HOW

它怎么做

  • 需要 embedding provider；缺失时返回 embedding_provider_missing，不阻塞其它 lane。
  • query text = user message + expanded domains + expanded keywords。
  • 候选文本和 lexical lane 一样来自 profile item 或 episode slice 的可检索字段。
  • 一次 embed query 和所有 records，计算 cosine similarity。
  • 低于 semantic.min_score 的候选过滤；默认阈值 0.58。
  • 按相似度降序取 semantic.top_k。
WHY

为什么需要它

旅行表达经常不是同一个词：用户说“别太折腾”，记忆里可能是“偏好轻松节奏”；用户说“自然口味”，记忆里可能是“不喜欢重油重辣”。Semantic lane 用向量相似度补足这类语义距离。

主要风险

语义召回可解释性弱，且会引入相似但不相关的候选。因此它只做候选生成，必须带 semantic_score 进入 evidence，后续由 fusion 和 Stage 4 reranker 收敛。

Fusion

多路候选不能简单拼接，需要合并和限流。
步骤
实现
设计理由
去重
按 item_id 合并同一候选。
同一条记忆可能被 symbolic 和 semantic 同时命中，应该增强证据而不是重复出现。
加权 RRF
weight / (rrf_k + rank)，默认 symbolic=1.0、lexical=0.6、semantic=0.8。
不同 lane 的原始分数不可直接比较，用排名融合更稳。
证据合并
合并 lanes、lane_ranks、lane_scores、matched keywords、destination match、semantic / lexical score。
让 Stage 4 能看到“为什么这条记忆被召回”。
排序 tie-break
先按 fused_score，再优先 symbolic，再按 source 和 item_id 稳定排序。
让结构化强证据在同分时更可信，同时保证结果可复现。
候选 cap
默认最多 30，总 profile 16，总 slice 16。
控制 Stage 4 reranker 成本，避免长尾噪声淹没有效候选。

开发者实现理由

为什么不是只做一种检索。
Determinism

先用规则拿高精度

旅行偏好里有大量硬约束，误召和漏召的代价不同。Symbolic lane 给出稳定、可解释的第一批候选。

Coverage

再用词面和语义补覆盖

用户语言和存储文本不会总是同词。Lexical 处理显性词面相似，Semantic 处理语义相似。

Observability

每条候选必须带证据

Stage 3 不直接决定最终使用哪条记忆，但它必须解释候选来源，方便调试 Stage 4 为什么选或不选。

为什么旅行场景需要多 lane

旅行表达经常口语化、跨领域、带目的地。
Aliases

同义表达很多

用户说“住哪里”，记忆里可能写“酒店”或“住宿”。关键词扩展能补齐这类差异。

Destination

目的地影响可迁移性

大阪美食经验和东京住宿经验不一定通用。Stage 3 会记录 destination alias / child / region 信息。

Evidence

候选需要证据

Stage 3 不只返回候选，还返回 evidence_by_id，给 Stage 4 判断为什么命中。

Telemetry

这些字段帮助区分“没查”“查了没命中”“命中了但被重排过滤”。
Policy

source_policy

requested_source、search_profile、search_slices、是否 widening。

Expansion

query_expansion

原始 domains/keywords、扩展后 domains/keywords、目的地别名。

Lanes

lanes_attempted

每条 lane 的候选数、成功/失败、fusion 前后候选数量。

Stage 3 有效的关键是“广一点拿候选，但保留证据”。真正的收敛交给 Stage 4 reranker。
